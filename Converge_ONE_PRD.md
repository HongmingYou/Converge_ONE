[WIP] ONE MVP PRD

一、背景
1. 产品定位： ONE 是一款融合「Agent 执行」与「个人创作」的 一站式 AI 工作台。 他将分散在各处的 AI 能力汇聚在统一场域，让用户可以指挥AI专家团队完成各类泛办公场景任务。
  1. 对于人类用户：ONE既是招之即来的超级团队，也是辅助工作的顺手工具。用户既能零门槛地驱动 AI 完成复杂交付，也能在"人主导"的深度办公场景下获得极简流畅的使用体验。
  2. 对于 Agents： 它是专家 Agent 能够稳定驻扎的“虚拟办公室”。ONE 在底层打通了 Agent 运行所需的上下文记忆、数据读写权限与交互环境，使 Agent 从“一次性对话”变为为工作流中的常驻角色。
2. 使命愿景： 探索并定义  人与 Agents Team 的协作范式，为「人 + Agents」的新型组织打造办公基础设施。
3. MVP 目标： 搭建基础产品框架，验证 ONE 作为「人 + Agents 协作工作台」的成立性，重点验证：
  - 调度价值：统一入口能否在不损失问答体验的前提下 有效调度现有垂直 Agent 能力？多 Agent 间的上下文打通能否产生协同效应
  - 共生价值：验证“人主导”的工作台体验，能否有效的承载用户“类知识库问答”的知识工作场景需求（总结会议纪要、总结播客等）

二、核心产品逻辑
1. work 方向和 AI 工作台都是代称。回归本质，我们最终希望 用户在生产力场景下需要用 AI 的时候就想到 ONE，所有泛办公的需求都能在 ONE 中找到解决方案，甚至成为每天办公时都要打开的产品；有这样野心的产品很多，离这个目标最近的有三大类：
  1. 大家目前办公时每天打开的软件，即现有办公产品(Workspace类产品)，如飞书、Notion、ClickUP
  2. 最被广泛使用的 ChatBot 产品：ChatGPT、Gemini
  1. 被塑造为 什么都能做的通用数字员工：Manus

2. 我们的要做完整的工作台 就需要覆盖 现有WorkSpace产品 的场景闭集，并将什么都能答的 ChatBot 和 什么都能干 Manus 有机融入进来；（本质在做的是 AI native 的飞书）
  1. 很多 IM 已有的功能抽象不需要变，需要变的是 Agent 作为数字员工进来之后功能内部形态
  2. 当“消息”需要处理的人与 Agent 的双向交互，第一界面就不再是信息流、而是统一对话入口来承载我的意图、审查 agent 结果。
  3. 当“知识”模块需要承载 agent 的工作产物，而且 ai 参与资产创作时，其交互界面也会发生变化。

  2. 但这些产品的底层架构（数据模型、权限体系）在 Day 1 是完全为人设计的。当它们试图引入 AI 时，会发现：
    - 产品交互范式以人的操作为核心，AI 难以获得足够的执行权限，AI 多是只能"建议"而非"执行"；
    - 传统 Workspace 的数据结构是为人类视觉优化的，而非为 LLM 语义理解设计的；Agent 难以理解复杂的业务上下文。
    - 现有系统只有“人”的权限系统，缺乏针对 Agent 的操作权限控制和结果审计；
  3. 这些不仅是技术问题，更是商业包袱。传统 Workspace 要解决这些问题则需要大刀阔斧，而这为 AI-Native 的新产品留下了结构性机会窗口。

4. Why We Can Win： Workspace 产品最终的归宿必然是平台化，但我们不能在 MVP 阶段去“补齐所有功能”，而是把长板做到最长， 我们的后发优势在于：
  1. vs 传统 Workspace  —— 无历史包袱的 AI-Native 架构
    - ChatGPT/Manus 已完成早期用户教育，用户开始接受通过 Chat 的方式描述需求，且接受"Agent 产出 80 分 + 人精修至 100 分"的工作模式；
    - Agent 产出物需要编辑容器，但容器不需要复杂功能——我们可以用更低成本开发"刚好够用"的编辑界面
    - 关键差异：没有产品上的历史包袱、我们可以在首屏就用 LUI 的方式承接用户的最宽泛意图；并在“数据存储、权限设置、界面功能”等维度，从 Day 1 就为「Agent 持续参与协作」预留接口；
  2. vs Manus  —— 垂直专家 > 通用选手
    - 我们已有打磨好的垂直场景能力：搜索（Hunter）、多媒体创作（Framia）、代码（Enter），这些是办公的基础能力，垂直专家 Agent 的整合效果应优于通用 Agent 的"什么都能做但都不够专业"
    - 关键差异：垂直产品持续优化专业深度 + 跨 Agent 上下文打通
  3. vs ChatGPT —— 不是所有工作都能在 Chat 界面承载
    - Chat + Canvas 结构适合"交付型任务"，但难以承载非线性"创作型工作"
    - ChatBot 无法吃掉全部工作场景，部分工作需要沉浸式的专属界面（类比 framia）
    - 关键差异：我们提供 Chat（交付型）+ User Workspace（创作型）双入口，覆盖完整的知识工作场景

5. 上述与竞品相比的核心差异也就是我们在 MVP 阶段要核心建设的功能

三、MVP 范围
1. 💬  Chat Interface: 
  1. Conversation：用户可以获得对标 ChatGPT/Gemini/Claude Web 的基础对话体验(按需 搜索问答)，支持多模型切换
  2. App Invocation ：用户可以手动连接「Framia、Enter」两种垂直应用并激活对应的产物生产链路、系统也可以针对典型 query 做对应应用的自动激活和任务派发
  3. Dynamic Canvas：垂直应用的产物在对话流旁的 Canva 内展示、 支持用户跳转到对应的专业产品容器内进行深度编辑
  4. Library： Chat 页面内产生的产物自动存放在 Library 中，用户可以溯源资产的产生过程和基于产物进行进一步的对话
  5. Projects：用户可以创建Projects、用户上传本地文档和URL到Project中、基于Project发起的对话会默认参考上传的文档进行通用对话，在Project页面也可以快速跳转到对应的Project编辑区进行沉浸式创作。
