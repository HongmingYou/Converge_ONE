一、背景

1. 产品定位

ONE 是一款融合「Agent 执行」与「个人创作」的 AI 交付工作台。它将分散在各处的 AI 能力汇聚在统一场域，将用户的自然语言意图转化成可交付、可复用的成果。

- 对于人类用户：它既是招之即来的超级团队，也是辅助工作的顺手工具。用户既能零门槛地驱动 AI 完成复杂交付，也能在"人主导"的深度办公场景下获得极简流畅的体验。
- 对于 Agents：它是专家 Agent 能够稳定驻扎的"虚拟办公室"。ONE 在底层打通了 Agent 运行所需的上下文记忆、数据读写权限与交互环境，使 Agent 从"一次性对话能力"升级为工作流中的常驻角色。

2. 使命愿景

探索并定义人与 Agents Team 的协作范式，为「人 + Agents」的新型组织打造办公基础设施。

3. MVP 目标

搭建基础产品框架，验证 ONE 作为「人 + Agents 协作工作台」的成立性。重点验证：
- 调度价值：统一入口能否有效调度现有垂直 Agent 能力？多 Agent 间的上下文打通能否产生协同效应（1+1>2）
- 共生价值：验证"人主导"的工作台体验，能否有效承载用户的知识工作场景需求（总结会议纪要、研究竞品报告等）


二、核心产品逻辑

### 1. 一句话结论：ONE 要解决的不是“让 AI 更会聊”，而是“让意图稳定变成可交付成果”

知识工作的本质是「意图 → 产出」。过去这条链路由人完成（理解、操作、加工、交付）；AI 的出现把“执行”从人手中可转移给 Agent，但要让它真正进入工作流，需要一个能承载“执行、上下文与沉淀”的工作台，而不只是对话框。

因此 ONE 的核心命题是：把自然语言变成可执行任务，把执行过程变成可控协作，把输出物变成可复用资产。

---

### 2. 为什么现在做：从 Copilot 走向 Agentic Execution，是一次产品范式切换

过去两年主流 AI 产品更多停留在“辅助建议”（Copilot）：写一段、补一句、给选项。它能提升效率，但很难对“交付”负责。

现在正在发生的变化是“可行动的智能”（Agentic Execution）：模型越来越擅长工具调用与多步任务。关键瓶颈随之转移——不是“模型会不会”，而是“产品能不能让它稳定执行、可控复盘、可沉淀复用”。

这意味着：新的赢家不只是在对话里更聪明，而是在工作流里更可靠。

---

### 3. 市场机会：传统 Workspace 的 AI 困境，本质是三条链路断裂

所有 Workspace 最终都会走向 All-in-One，也都在积极拥抱 AI。但它们的起点决定了基因：Day 1 为“人操作工具”设计，AI 只能挂在边上，变成“建议层”。当它们试图进入“执行层”时，会遇到三条结构性断裂：

| 断裂 | 表现 | 根因 |
|------|------|------|
| **意图 → 执行断裂** | AI 只能建议，难以端到端交付；人仍需在多个工具间串联 | 权限与操作路径围绕人设计，AI 缺少稳定执行与闭环机制 |
| **执行 → 上下文断裂** | 跨模块信息不通，任务一长就丢上下文；多 Agent 难协同 | 数据结构按模块割裂，缺乏统一“上下文层/记忆层” |
| **输出 → 沉淀断裂** | 产出即抛，难以复用与管理；难形成资产与组织记忆 | 没有为“AI 产出物”设计统一存储、版本、引用与再加工机制 |

核心判断：这不是功能缺失，而是架构债务。传统 Workspace 若要跨过这三条断裂，需要重构底层——这为 AI-Native 的新产品留下结构性窗口。

---

### 4. 用户与任务：知识工作天然分为两种模式，需要两种入口但共享同一底座

我们把知识工作分为两类，不是“新手/高级”，而是任务性质不同：

| 维度 | 交付型工作 | 创作型工作 |
|------|-----------|-----------|
| **本质** | 有明确交付物、可验收的任务 | 需要长时间沉浸的整理与创作 |
| **典型任务** | 做竞品分析报告、写代码、做海报/视频 | 总结会议纪要、制作 PPT、撰写长文档 |
| **用户心智** | “帮我做” | “我来主导，你来辅助” |
| **AI 角色** | 执行者（Agent 主导，人审核） | 助手（人主导，AI 参与） |
| **最优交互** | 对话驱动任务编排 + 过程可视化 | 沉浸式编辑界面 + 侧边 AI |
| **产品入口** | Chat Interface | User Workspace |

结论：需要“双入口”（Chat / Workspace），但必须共享同一套上下文与资产底座，否则会再次产生割裂。

---

### 5. 设计原则：用四条原则把“范式切换”落到产品形态上

1) **Language-first（语言优先）**：把语言当作最高层控制面，用于表达目标、约束、偏好与验收标准。  
2) **Execution-first（执行优先）**：产品必须让 AI 真正“做事”，对执行过程可观测、可纠错、可回滚。  
3) **Context-unified（上下文统一）**：跨 Agent、跨模块共享同一上下文层（任务、素材、引用、决策与中间产物）。  
4) **Asset-native（资产原生）**：默认把 AI 产出当作“资产”而非“聊天记录”，可管理、可引用、可二次加工。  

---

### 6. 产品架构：三层结构（入口层 / 工作台层 / 上下文层）对应三条断裂

```
┌─────────────────────────────────────────────────────────────────────────┐
│                           CONVERGE ONE                                   │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│  ┌───────────────────────────────┐  ┌───────────────────────────────┐  │
│  │  Chat Interface               │  │  User Workspace                │  │
│  │  （交付型入口）                │  │  （创作型入口）                │  │
│  │  意图表达 / 调度 / 汇报        │  │  文件树 + 编辑区 + 侧边 AI      │  │
│  └───────────────────────────────┘  └───────────────────────────────┘  │
│                    ↓                           ↓                         │
│          ┌───────────────────────────┐  ┌───────────────────────────┐   │
│          │ Artifact Canvas            │  │ ONE's Workspace            │   │
│          │（交付物预览/迭代）         │  │（产出物归档/复用）          │   │
│          └───────────────────────────┘  └───────────────────────────┘   │
│                                                                         │
│  ┌─────────────────────────────────────────────────────────────────┐   │
│  │  Context Bus：跨 Agent 上下文共享层（任务/素材/引用/中间产物）      │   │
│  └─────────────────────────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────────────────────────┘
```

**A. Chat Interface（交付型工作入口）**
- **定位**：Agent 的控制台与汇报窗口  
- **核心链路**：意图表达 → @mention 调度 → Agent 执行 → Canvas 预览/迭代 → 归档  
- **关键能力**：多 Agent 调度、执行过程可视化、输出可迭代、上下文共享  

**B. ONE's Workspace（AI 产出中心）**
- **定位**：Agent 产出的统一归档地  
- **核心价值**：让产出从“回复”升级为可沉淀、可复用的“数字资产”  
- **支持类型**：Image / Code / Document / Workflow（MVP 可先覆盖最常见的 2-3 类）  

**C. User Workspace（创作型工作入口）**
- **定位**：人主导的深度创作工作台  
- **统一界面框架**：左侧文件树 + 中间编辑区 + 右侧 AI Chat/Agent  
- **MVP 首先实现**：Knowledge Base（类 NotebookLM 知识工作台）  
  - Sources：文档上传与知识索引  
  - Chat：基于知识库的 RAG 问答  
  - Notes：人工笔记/精修编辑  
  - Outputs：一键生成 Slides / Summary / Report（强调“可编辑的输出物”）  

---

### 7. MVP 策略：链路通 > 功能全（验证两个核心假设）

在投入大量资源前，我们需要验证两个核心假设：

| 假设 | 验证问题 | 如果成立 |
|------|---------|---------|
| **调度假设** | 统一入口调度多个垂直 Agent，能否产生 1+1>2 的协同效应？ | ONE 作为“超级入口”有独立价值 |
| **共生假设** | 人主导的工作台 + AI 辅助，能否有效承载高频知识工作场景？ | ONE 不只是 Agent 调度器，也是创作工具 |

因此 MVP 聚焦打通两条最能体现差异化的端到端链路：

- **链路 A（交付链路）**：Chat → Canvas → 归档  
  目标：让用户在一次会话里完成跨能力的交付（研究→生成→修订→归档）。  
- **链路 B（创作链路）**：Workspace → 知识整理 → 生成输出  
  目标：让用户愿意把资料放进来反复使用，输出能继续编辑与复用。  

---

### 8. MVP 边界（确保资源聚焦）

| 做 | 不做 |
|----|------|
| ✅ Chat + Canvas + 归档链路 | ❌ CRM / 客户管理 |
| ✅ @mention 调度 Hunter/Framia/Enter | ❌ 日历 / 日程集成 |
| ✅ Context Bus 跨 Agent 上下文共享 | ❌ 审批流 / 工作流引擎 |
| ✅ Knowledge Base 知识工作台 | ❌ 复杂的团队协作与权限 |
| ✅ 统一的 User Workspace 框架 | ❌ My AI Apps（MVP 后考虑） |

---

### 9. MVP 验证指标（让团队对“做成什么样算成功”有共识）

| 验证维度 | 核心问题 | 成功标准（初版） |
|---------|---------|----------------|
| **调度价值** | 统一入口能否有效调度 Hunter/Framia/Enter？ | 用户能在一个 Chat 中完成跨 Agent 任务 |
| **协同价值** | 多 Agent 上下文打通能否产生 1+1>2？ | Hunter 结果可被 Framia/Enter 直接引用，减少重复输入与返工 |
| **体验价值** | 类 NotebookLM 体验能否满足知识工作场景？ | 用户愿意在 Knowledge Base 中完成“总结/生成”类任务并反复回访 |
| **框架扩展性** | 统一界面框架能否承载多种创作场景？ | 后续 PPT 编辑器/文档编辑可复用同一框架 |

**北极星指标（初版）**：WAU 中完成至少一个可归档 Artifact 的用户占比 > 60%
